# 工作流最佳实践

> 保留原 guides 的方法论，但在 OpenCode 下重新组织执行路径

**版本**: v1.3
**适用**: OpenCode（2026 年）

---

## 1. 核心开发循环

OpenCode 版仍采用同一个方法论：

```text
Explore -> Plan -> Code -> Verify -> Simplify -> Commit
```

差别不在方法论，而在执行媒介：
- Explore 更多借助 `explore` subagent
- Plan 更多借助 `plan` agent
- Code 主要由 `build` agent 实施
- 自动化由 plugins/formatters/permissions 承担

---

## 2. 复杂度分流

### 简单任务

判断标准：
- 一句话说清
- 改动范围小
- 不涉及架构讨论

流程：

```text
Code -> Verify -> Commit
```

### 中等任务

判断标准：
- 改多个文件
- 需要先了解上下游
- 方向基本明确

流程：

```text
Explore -> Plan -> Code -> Verify -> Simplify -> Commit
```

建议：
- Explore 和 Diagnose 类型命令优先作为 subtask 运行
- Plan 输出应尽量落到 spec 或 session-notes，而不是只留在聊天历史里

### 复杂任务

判断标准：
- 需求需要多轮讨论
- 涉及模块边界或架构变更
- 预计跨多个会话

流程：

```text
讨论 -> /spec -> Plan -> Phase 1 实施 -> /handoff -> 继续下一阶段
```

## 3. Spec 驱动实施

复杂功能 SHOULD 先写 spec。

推荐结构：

```markdown
# [功能名]

## 背景与目标
## 需求概要
## 设计方案
## 风险与边界
## Implementation Phases
### Phase 1
- [ ] Task A
- [ ] Task B
Gate:
- [ ] 测试通过
- [ ] 无回归
```

实施时：
- 每完成一个 task 就勾选
- 每完成一个 phase 就检查 gate
- phase 结束后更新 roadmap

---

## 4. Roadmap 驱动进度管理

复杂项目必须有 `docs/roadmap/`。

### 建议结构

```text
docs/roadmap/
├── README.md
├── phase-1-xxx.md
├── phase-2-xxx.md
└── phase-3-xxx.md
```

### 使用规则

- `README.md` 只写全局总览
- 当前 phase 单独成文
- 已完成 phase 不必频繁加载

---

## 5. 会话治理

OpenCode 有 compaction 和 snapshot，但这不等于完整的项目记忆系统。

因此要养成三种习惯：

### 1. 状态落盘

长任务的真实状态 MUST 写进：
- spec
- roadmap
- session-notes

如果任务跨多个阶段，SHOULD 再补一个独立的计划文件或 spec phase 清单。

### 2. 主动交接

不要等上下文完全混乱才总结。

推荐语：

```text
执行 handoff，更新当前 spec 和 roadmap，并把下一步写到 session-notes。
```

### 3. 恢复时先读磁盘

恢复任务时先读取：
- `AGENTS.md`
- roadmap
- 活跃 spec
- `session-notes`

## 6. 代码质量防御

迁移到 OpenCode 后，质量防御建议分三层：

### 第一层：规则

在 `AGENTS.md` 中写清红线：
- MUST 先探索再改
- MUST 提交前验证
- MUST NOT 读敏感文件

### 第二层：配置

在 `opencode.json` 中限制：
- `plan` agent 禁止编辑
- `bash/edit/webfetch` 默认 ask
- formatter 自动整理代码

### 第三层：命令与插件

- `/done` 做收尾检查
- `/release` 做 phase 级刷新
- plugin 拦截危险读取和危险 shell
- compaction plugin 补充默认摘要容易漏掉的项目状态

---

## 7. 并行会话与多 Agent

OpenCode 支持多会话、子 agent、共享链接。

推荐使用方式：

### 会话分工

- 会话 A：主线实现
- 会话 B：独立探索或审查
- 会话 C：文档梳理或验证

### Agent 分工

- `plan`: 分析任务边界
- `explore`: 做深度阅读
- `build`: 执行改动
- `general`: 处理轻量 side task
- `review`: 只读审查
- `docs-writer`: 文档整理
- `security-auditor`: 安全和敏感变更审计

原则：
- 阻塞主线的问题优先在主会话解决
- 非阻塞探索可以并行

## 8. 文档体系

推荐保留原 guides 的文档体系：

```text
docs/
├── roadmap/
├── specs/
├── architecture/
└── development/
```

### `docs/architecture/`

写项目结构、模块划分、数据流。

### `docs/development/`

写上手、部署、changelog。

### `docs/specs/`

写重要功能设计与阶段计划。

### `docs/roadmap/`

写项目阶段性进展。

### 社区实践补充

对于长期项目，一个更稳的模式是：
- `AGENTS.md` 只保留导航和硬约束
- 将产品目标、架构原则、设计语言拆成独立文档
- 通过 `instructions` 或按需读取把它们组合回来

再往前一步，推荐再补：
- `docs/architecture/adr/` 记录关键决策
- `docs/roadmap/_phase-template.md` 统一 roadmap phase 格式
- `docs/specs/_template.md` 统一 spec 结构

---

## 9. 推荐的完成标准

每个功能完成时至少检查：

- 测试通过
- lint 通过
- 边界场景已检查
- 涉及的 spec 已同步
- 涉及的 roadmap 已同步
- 需要交接时已写 `session-notes`

---

## 10. 反模式

避免这些迁移误区：

- 试图把 Claude Hooks 原样搬到 OpenCode
- 把所有工作流都写成一个超大 `AGENTS.md`
- 依赖会话记忆而不落盘
- 把 commands 和 skills 混成一类东西
- 给所有 agent 全权限
