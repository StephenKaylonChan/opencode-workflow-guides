# 维护约定

## 修改流程

1. 先确定这次改动影响的是：
   - 主指南
   - 模板
   - 两者同时
2. 先做必要的探索和调研
3. 修改文件
4. 执行一致性检查
5. 更新版本记录和 roadmap

## 一致性检查清单

- `README.md` 与 `00-04` 的版本号是否一致
- 新增的 OpenCode 能力是否同时更新了主文档和 `templates/base-project/`
- roadmap phase 是否与实际进展一致
- 是否误用了 Claude Code 专属术语

## 推荐命令

- `/catchup`：恢复当前维护上下文
- `/audit`：检查版本同步、roadmap 和模板一致性
- `/docs`：刷新仓库自身文档
- `/release`：检查是否需要发起一轮版本升级
- `/review`：用只读 agent 做问题导向审查
- `/test`：执行文档一致性检查
- `/version-sync`：协助处理版本号同步

## 建议的开工顺序

每次新开会话，默认顺序是：

1. `/catchup`
2. 看 roadmap 当前 phase
3. 判断本轮是文档、模板还是调研工作
4. 动手前如有必要先 `/audit`

## 版本规则

- 影响主文档或模板行为的小功能更新：小版本 +1
- 仅错别字或轻微措辞修正：可不升级版本，除非会影响理解

## 何时更新 roadmap

- 完成一个明确阶段时
- 开始一个新的维护阶段时
- 发现当前路线图与实际工作偏离时

## 何时写 research

- 需要联网调研官方新能力时
- 需要吸收社区实践时
- 做出可能影响长期结构的判断时
