## 1. 第一次：新项目初始化

第一次进入项目时，让 OpenCode 读取本目录文档并完成初始化：

```text
请阅读 ~/Downloads/00_project/guides-opencode/ 下的所有主文档，
然后根据我的项目完成 OpenCode 协作体系初始化。
技术栈是：[在这里写实际技术栈]
```

初始化目标：
- 创建 `AGENTS.md`
- 创建 `opencode.json`
- 创建 `.opencode/commands/`
- 创建 `.opencode/skills/`
- 创建 `.opencode/plugins/`
- 创建 `docs/roadmap/` 和 `docs/specs/`
- 创建 `.opencode/session-notes.md`

## 2. 日常开发流程

### 标准节奏

```text
你提出需求
  ->
OpenCode 先探索相关代码
  ->
必要时切到 plan agent 形成方案
  ->
build agent 实施
  ->
验证测试、lint、边界场景
  ->
需要时做简化审查
  ->
提交
```

推荐提示语：

```text
先不要改代码，先探索相关文件，告诉我当前结构和可选方案。
```

```text
把方案写成一个简短计划，再按计划实施。
```

### 小任务

方向明确、影响面小的任务直接做，不必先写 Spec：

```text
帮我处理这个小任务：[一句话描述]
```

### 复杂任务

复杂任务先讨论，再落盘为 Spec：

```text
先分析需求和方案，不要写代码。讨论完成后整理到 docs/specs/xxx.md。
```

然后再开始实施：

```text
读取 docs/specs/xxx.md，开始实施 Phase 1。
```

## 3. 路线图与 Spec

### 路线图

`docs/roadmap/` 负责记录项目整体阶段和当前聚焦项。

建议只在 `AGENTS.md` 或 `opencode.json.instructions` 中加载：
- `docs/roadmap/README.md`
- 当前进行中的 phase 文件

不要把全部历史 roadmap 文件都塞进上下文。

### Spec

`docs/specs/` 负责记录功能设计、阶段拆分、完成标准。

简单任务：
- 不必建 spec

需要多轮讨论、跨多天实施、涉及架构设计的任务：
- MUST 建 spec

## 4. 会话和上下文管理

OpenCode 有 compaction、snapshot、share，但没有 Claude Code 那种等价的 Auto Memory 体系。

因此推荐的上下文策略是：

### 70% 规则

- 会话进行到明显变长时，主动总结当前状态
- 将当前进度写入 `.opencode/session-notes.md`
- 长任务必须同步勾选 `docs/specs/` 里的 tasks
- 路线图变化同步写入 `docs/roadmap/`

### 会话切换前

至少落盘以下三类信息：
- 当前正在做什么
- 已完成什么
- 下一步是什么

### 推荐交接语

```text
请把当前进度整理到 .opencode/session-notes.md，
并更新 roadmap/spec 中已完成的部分。
```

### 分析型命令尽量用 subtask

像 `/catchup`、`/docs`、`/diagnose`、`/release` 这类偏分析和整理的命令，推荐配置 `subtask: true`。

这样做的好处：
- 不污染主会话上下文
- 适合长输出和扫描型任务
- 主线实现会话更干净

## 5. Commands、Skills、Agents 的区别

### Commands

放在 `.opencode/commands/`。
适合显式触发的工作流，如：
- `/handoff`
- `/catchup`
- `/spec`
- `/task`
- `/audit`
- `/review`
- `/test`
- `/done`

### Skills

放在 `.opencode/skills/`。
适合被 command 或 agent 按需加载的可复用能力块，如：
- 文档刷新规则
- 诊断方法
- 审计检查表

### Agents

OpenCode 的内置 agent 重点是：
- `build`: 执行开发
- `plan`: 规划与分析
- `explore`: 深度探索
- `general`: 通用子任务

项目级还可以补自定义 agents，例如：
- `review`: 只读审查
- `docs-writer`: 文档整理
- `security-auditor`: 安全检查

简单理解：
- **想做一件事** -> command
- **想复用一套做法** -> skill
- **想切一种工作模式** -> agent

## 6. 常见场景建议

### 场景：开始一个复杂需求

```text
先探索相关代码和文档，不要写代码。
给我 2 个方案，比较风险和改动面。
必要时切 plan agent。
```

### 场景：需求讨论跨多个会话

```text
把当前结论整理到 docs/specs/xxx.md，
列出已确认、未确认、下一步待讨论的问题。
```

### 场景：准备结束本次会话

```text
执行一次 handoff：
1. 更新 roadmap/spec
2. 写 session-notes
3. 总结下一步
```

### 场景：恢复工作

```text
读取 AGENTS.md、docs/roadmap/README.md、当前 phase、
活跃 spec 和 .opencode/session-notes.md，
告诉我现在做到哪里了。
```
